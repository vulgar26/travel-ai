# Travel AI Planner — 产品化与简历级升级方案

本文档将项目从「学习 / Demo」推进到**可写进简历、具备可讲清楚的亮点、并具备上线最小集（MVP）**的目标，给出**分阶段任务、验收标准、工时预估与风险**。  
适用范围：当前技术栈（Spring Boot 3、Spring AI Alibaba、PostgreSQL + pgvector、Redis、SSE）。

---

| 用途 | 说明 |
|------|------|
| **个人路线图** | 按阶段勾选任务，控制范围，避免功能堆砌。 |
| **简历素材** | 每阶段末的「简历可写表述」可直接改编为项目经历 bullet。 |

**阅读约定**：工时按**单人、兼职或全职混合**估算为「人周」量级；实际以你的周投入为准上下浮动。

## 2. 当前基线与目标画像

### 2.1 当前项目已有能力（优势）

- Spring Boot 3 + Spring AI（通义）对话与流式输出（SSE）。
- RAG 相关探索：查询改写、向量检索、知识上传、自研 `VectorStore`（JDBC + pgvector）。
- Redis 对话记忆、工具调用（天气）占位或部分实现。
- 文档与代码在包名、配置、实现细节上存在不一致，需统一后再对外叙事。

### 2.2 「简历级 + 亮点」目标定义

**一句话定位（可放在简历项目栏）：**

> 基于 Spring AI 与 RAG 的智能出行助手后端：多路查询改写、pgvector 知识库、工具调用与 SSE 流式输出；完成鉴权、限流、可观测与容器化部署，支持多环境配置与自动化测试。

**「亮点」应落在可验证的工程事实，例如：**

- RAG **单一路径**清晰、无重复检索浪费；有**自动化测试**防止回归。
- **密钥与配置**不进入版本库；**用户与会话**绑定，避免越权。
- **一键部署**（Docker Compose 或等价），含健康检查与基础监控/日志规范。
- （可选）知识库 **metadata / 多租户**、成本与缓存策略等**可展开架构题**。

### 2.3 「产品级可上线」最小定义（本文采纳）

满足以下即视为达到**可对外演示 / 内测上线**的「产品级 MVP」（非商业化完备）：

1. 安全：无明文密钥入库；接口有鉴权；敏感路径有基本防护。  
2. 可靠：外部依赖（LLM、天气、DB）**超时与失败可预期**，不全站拖死。  
3. 可运维：健康检查、结构化日志、可区分环境。  
4. 可重复：`mvn test` 或通过 CI 构建镜像；README 可复现启动。

## 3. 差距分析（从现状到目标）

| 维度 | 典型现状风险 | 产品级目标 | 优先级 |
|------|----------------|------------|--------|
| **RAG 链路** | 手工检索与 Advisor 重复；上下文未正确注入模型 | 单一明确链路；行为与 README 一致；可测 | P0 |
| **密钥与配置** | API Key、密码写入仓库 | 环境变量 / 本地覆盖文件 + `.gitignore`；文档仅占位符 | P0 |
| **鉴权与多租户** | `conversationId` 可枚举猜测 | 用户身份与会话绑定；禁止越权读历史 | P0～P1 |
| **上传与向量** | 无类型/大小约束；启动灌库可能重复 | 限制与校验；灌库幂等或可配置开关 | P1 |
| **外部调用** | 天气等可能为 Mock 或未接完 | 真实调用 + 降级文案，或文档明确 Mock | P1 |
| **可观测** | `System.out` 为主 | SLF4J、请求 ID、可选 Metrics/Tracing | P1～P2 |
| **部署** | 本地手工启动 | Dockerfile + Compose；可选 CI | P1 |
| **合规与说明** | 未声明 | README 说明数据流、演示边界与免责声明 | P2 |

## 4. 分阶段实施方案（详细）

### 阶段 0：基线冻结（0.5 天）

**目标**：固定「当前仓库」为改造前快照，便于对比与回滚。

| 任务 | 说明 |
|------|------|
| 打 tag 或分支 | 如 `pre-product-pass` |
| 记录运行环境 | JDK 21、PostgreSQL 版本、Redis、通义模型名 |

**验收**：能从 README/本机笔记复现一次启动（即使仍有技术债）。

---

### 阶段 1：技术债清零与叙事成立（约 1～2 周）

**目标**：代码行为与文档一致；RAG 逻辑**闭环**；为后续改动加**测试安全网**。

#### 4.1.1 RAG 架构决策（必选其一，禁止长期双轨）

| 方案 | 做法概要 | 优点 | 注意 |
|------|----------|------|------|
| **A. 去 Advisor，保留手写管线** | 移除 `RetrievalAugmentationAdvisor`，仅用「改写 → 多路检索 → 拼上下文 → `ChatClient.prompt(含上下文的用户消息)`」 | 与 README「多路改写」叙事一致；易单测拼接结果 | 需在代码中明确最终 prompt 形态 |
| **B. 保留 Advisor，自定义 Retriever** | 实现/封装「多 query 合并检索」的 Retriever，仅一处检索 | 符合 Spring AI  Advisor 模式 | 实现成本略高 |

**验收标准**：

- [ ] 单次用户请求路径中，**向量检索逻辑只有一套**（或合并为一次批量检索），避免重复计费与延迟。
- [ ] 检索到的文本**确实进入**发给模型的 user/system 内容（可通过 debug 日志或集成测试断言）。
- [ ] README「多路召回」描述与实现一致。

#### 4.1.2 密钥与配置

| 任务 | 说明 |
|------|------|
| 外置敏感配置 | `spring.ai.dashscope.api-key`、`weather.api-key`、数据库密码等使用 `SPRING_AI_DASHSCOPE_API_KEY` 等环境变量或 `application-local.yml` |
| 版本库 | `.gitignore` 增加 `application-local.yml`、`*.local.yml` 等 |
| 文档 | `application.yml` 示例仅含占位符；**已提交过的真实密钥必须轮换（作废并重新生成）** |

**验收标准**：

- [ ] `git grep` 在跟踪文件中搜不到真实 Key 形态（或仅示例占位符）。
- [ ] 新同事仅按 README + 环境变量可启动。

#### 4.1.3 天气工具

| 任务 | 说明 |
|------|------|
| 真实接入 | 和风 / QWeather HTTP 调用；解析 JSON；超时与异常处理 |
| 或明确 Mock | 若暂不接 API：README 与接口响应中明确「演示用模拟数据」 |

**验收标准**：

- [ ] 行为与文档一致；异常时不导致 SSE 整条链路 500 无提示。

#### 4.1.4 测试

| 类型 | 内容 |
|------|------|
| 集成测试 | RAG 拼接逻辑：Mock `EmbeddingModel` / `VectorStore` 或 Testcontainers（可选），断言最终传入 `ChatClient` 的内容包含检索片段 |
| 单元测试 | `QueryRewriter` 输出解析（可 Mock LLM 返回固定字符串） |

**验收标准**：

- [ ] `mvn test` 在 CI 或本地可持续通过。

**简历可写表述（示例）**：

- 「梳理 Spring AI Advisor 与手写 RAG 的重复检索问题，统一为单一检索链路，并补充集成测试防止回归。」
- 「将密钥与配置外置，消除仓库明文敏感信息并轮换已泄露凭证。」

---

### 阶段 2：可上线最小安全与运维包（约 2～3 周）

**目标**：可部署到公网或隔离网络演示；具备基本**防误用、防拖垮**能力。

#### 4.2.1 认证与授权

| 任务 | 说明 |
|------|------|
| 身份 | JWT（无状态）或 Session（有状态）；登录/注册可极简（甚至固定测试账号仅用于 Demo） |
| 会话绑定 | `conversationId` 仅允许所属用户访问；服务端生成 UUID，不信任客户端随意指定敏感 ID |
| 可选 | Spring Security 过滤器链统一处理 |

**验收标准**：

- [ ] 未携带合法凭证无法访问对话接口。
- [ ] 用户 A 无法通过修改路径参数读取用户 B 的 `conversationId` 历史。

#### 4.2.2 API 与输入安全

| 任务 | 说明 |
|------|------|
| 全局异常 | `@ControllerAdvice` 统一错误体：`code`、`message`、可选 `traceId` |
| 上传 | `MultipartFile` 限制扩展名（如 `.txt`）、大小上限（如 5MB）、编码 UTF-8；拒绝可执行内容 |
| 限流 | Bucket4j 或 Spring Cloud Gateway / Nginx 限流；按 IP 或用户 ID |

**验收标准**：

- [ ] 超大文件上传返回 413/400，而非 OOM。
- [ ] 高频请求触发限流时有明确 HTTP 状态与 Retry-After（可选）。

#### 4.2.3 韧性

| 任务 | 说明 |
|------|------|
| 超时 | `RestTemplate`/`WebClient`/OkHttp 为天气与第三方调用设置 connect/read timeout |
| LLM | 配置 Spring AI 侧超时；SSE 断开时资源释放 |
| 降级 | 检索失败时仍允许模型基于通用知识回答（可选开关），并日志记录 |

#### 4.2.4 可运维

| 任务 | 说明 |
|------|------|
| Actuator | `spring-boot-starter-actuator`：`/actuator/health`（含 Redis、DB 自定义 indicator 可选） |
| 日志 | SLF4J；MDC 注入 `requestId`；生产不打印完整 prompt/PII |
| 容器 | 多阶段 Dockerfile；`docker-compose.yml`：app + postgres + redis |

**验收标准**：

- [ ] 新环境 `docker compose up` 后健康检查通过即可认为「服务就绪」。
- [ ] 日志可 grep 请求链路。

**简历可写表述（示例）**：

- 「引入 JWT 与会话级授权，修复对话 ID 可枚举导致的越权风险。」
- 「统一异常处理与上传校验，集成 Actuator 健康检查并完成 Docker Compose 一键编排。」

---

### 阶段 3：差异化亮点（可选，约 3～4 周，选 2～3 项深挖）

**原则**：宁可把 2 个方向做深，也不要列 10 个浅层功能。

#### 4.3.1 知识库工程化

| 任务 | 说明 |
|------|------|
| Metadata | 文档块写入 `userId`、`sourceFile`、`uploadedAt`；检索时带 filter |
| 生命周期 | 列表、删除、按文件删除向量；避免 `KnowledgeInitializer` 每次启动重复灌库 |
| 迁移 | Flyway/Liquibase 管理 `vector_store` 表结构 |

**简历表述**：「设计向量 metadata 与按租户/用户隔离的检索策略，并实现知识块生命周期管理。」

#### 4.3.2 成本与体验

| 任务 | 说明 |
|------|------|
| 缓存 | 同一 session 相似问题短时缓存改写结果（Caffeine/Redis） |
| Prompt | 控制检索片段总长度；避免超长 context |

**简历表述**：「通过缓存与上下文裁剪降低重复调用大模型与 Embedding 的成本。」

#### 4.3.3 可观测性进阶

| 任务 | 说明 |
|------|------|
| Metrics | Micrometer：QPS、LLM 延迟、检索耗时 |
| Tracing | OpenTelemetry + Jaeger（可选） |

**简历表述**：「接入 Micrometer 指标，区分检索与生成阶段耗时，支撑性能优化。」

---

### 阶段 4：上线与合规（与阶段 2～3 重叠、持续）

| 任务 | 说明 |
|------|------|
| CI | GitHub Actions / GitLab CI：`mvn -B test`、构建镜像、推仓库（可选） |
| 环境 | `dev` / `staging` / `prod` 分配置；禁止 prod debug 打印敏感内容 |
| 域名与合规 | 若公网面向国内用户：了解 ICP、隐私政策；最小化 README 中「数据用途与保留策略」说明 |
| 备份 | PostgreSQL 定期快照；Redis 对话记忆是否持久化策略写清 |

## 5. MVP 清单（建议作为发布前 Gate）

完成下列项即可在简历写「已完成生产化改造并支持容器化部署」类描述：

- [ ] **RAG 单链路**，检索结果进入模型；测试覆盖关键路径  
- [ ] **无明文密钥**于仓库；**已泄露 Key 已轮换**  
- [ ] **鉴权** + 会话与用户绑定  
- [ ] **Docker Compose** 或等价一键启动  
- [ ] **Actuator 健康检查** + 结构化日志（含 requestId）  
- [ ] **README**：架构图（一张）、环境变量表、启动步骤、**已知限制**  
- [ ] （加分）30～60 秒**演示录屏**或截图：上传文档 → 对话 → 流式输出

## 6. 工时与可行性（粗估）

| 阶段 | 人周（单人） | 说明 |
|------|----------------|------|
| 阶段 1 | 1～2 | 含测试与文档；若 Testcontainers 则偏上 |
| 阶段 2 | 2～3 | 含安全、Compose、Actuator |
| 阶段 3 | 3～4 | 仅选 2～3 个方向时酌减 |
| 阶段 4 | 持续 | CI/合规随项目并行 |

**结论**：

- **约 4～6 人周**可达到「简历 + 可演示上线 MVP」的强叙事（阶段 1 + 2 + MVP 清单）。  
- **阶段 3** 用于面试「架构与权衡」加分，非门槛。

## 7. 风险与应对

| 风险 | 应对 |
|------|------|
| 范围膨胀 | 严格按 MVP 清单；新需求记入 backlog，不混入当前阶段 |
| 大模型费用 | 开发环境用较小模型或限流；缓存改写与检索结果 |
| 向量库与业务库同一 PostgreSQL | MVP 可接受；压力大时读写分离或专用向量实例 |
| 一人维护 | 自动化测试 + 文档即「可交接」资产 |

## 9. 面试可展开点（与实现对应）

1. **为何去掉双重 RAG / 为何选 Advisor 或手写**：成本、延迟、可测性。  
2. **Redis 记忆 vs DB**：TTL、条数截断、是否需要长期归档。  
3. **pgvector 自研 vs 官方 starter**：学习价值、维护成本、何时迁移。  
4. **SSE 与超时**：客户端断开、服务端背压。  
5. **上线 checklist**：健康检查、密钥、日志脱敏、限流。
